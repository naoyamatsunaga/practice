# モデル層の実装

## モデル層とは

モデル層（Model Layer）は、アプリケーションのビジネスロジックやデータ構造を表現する層です。UI層（View）やデータアクセス層（Database）から独立して、アプリケーションの中核となるデータ構造を定義します。

## ActivityModelの実装

このプロジェクトでは、`ActivityModel`クラスをモデル層として実装しています。

```dart
class ActivityModel {
  const ActivityModel({
    required this.id,
    required this.points,
    required this.title,
    required this.description,
    required this.createdAt,
    required this.updatedAt,
  });

  factory ActivityModel.fromActivityPoint(ActivityPoint activityPoint) {
    return ActivityModel(
      id: activityPoint.id,
      points: activityPoint.points,
      title: activityPoint.title,
      description: activityPoint.description,
      createdAt: activityPoint.createdAt,
      updatedAt: activityPoint.updatedAt,
    );
  }

  ActivityPoint toActivityPoint() {
    return ActivityPoint(
      id: id,
      points: points,
      title: title,
      description: description,
      createdAt: createdAt,
      updatedAt: updatedAt,
    );
  }

  final int id;
  final int points;
  final String title;
  final String description;
  final DateTime createdAt;
  final DateTime updatedAt;
}
```

## モデル層を実装するメリット

### 1. **関心の分離（Separation of Concerns）**

UI層やデータベース層から独立したモデルを定義することで、各層の責務が明確になります。

- **UI層**: 画面表示に専念
- **モデル層**: ビジネスロジックとデータ構造を管理
- **データベース層**: データの永続化に専念

### 2. **データベース実装の変更に強い**

データベースライブラリ（Drift、SQLite、Firebase等）を変更しても、モデル層のインターフェースを維持すれば、UI層への影響を最小限に抑えられます。

**例**: DriftからFirebaseに移行する場合
- モデル層があれば: `ActivityModel`のインターフェースは変更せず、変換ロジックだけ修正
- モデル層がなければ: UI層のコード全体を修正する必要がある

### 3. **ビジネスロジックの集約**

データに関する計算やバリデーションなどのロジックをモデル層に集約できます。

**例**: 将来的に以下のような機能を追加する場合
```dart
// ActivityModelに追加できる機能
int get totalPoints => points * multiplier;
bool get isExpired => updatedAt.difference(DateTime.now()).inDays > 30;
String get formattedDate => DateFormat('yyyy/MM/dd').format(createdAt);
```

### 4. **テストの容易性**

モデル層はデータベースやUIに依存しないため、単体テストが書きやすくなります。

```dart
// テスト例
void main() {
  test('ActivityModel should convert from ActivityPoint correctly', () {
    final point = ActivityPoint(
      id: 1,
      points: 100,
      title: 'テスト',
      description: '説明',
      createdAt: DateTime.now(),
      updatedAt: DateTime.now(),
    );
    
    final model = ActivityModel.fromActivityPoint(point);
    expect(model.id, 1);
    expect(model.points, 100);
    expect(model.title, 'テスト');
  });
}
```

### 5. **型安全性の向上**

モデル層を定義することで、コンパイル時に型チェックが行われ、実行時エラーを防げます。

### 6. **再利用性の向上**

同じモデルを複数の画面や機能で再利用できます。データ構造が一貫しているため、コードの重複を減らせます。

## 実装パターン

### ファクトリメソッドパターン

`fromActivityPoint()`のようなファクトリメソッドを使用することで、異なるデータソースからモデルを生成できます。

```dart
// データベースから
ActivityModel.fromActivityPoint(activityPoint)

// 将来的にAPIからも
ActivityModel.fromJson(jsonData)
```

### 変換メソッド

`toActivityPoint()`のような変換メソッドにより、モデルからデータベースエンティティへの変換が容易になります。

## 注意点

1. **過度な抽象化を避ける**: シンプルなアプリでは、モデル層が不要な場合もあります
2. **変換コスト**: データベースエンティティとモデルの間で変換を行うため、パフォーマンスへの影響を考慮する
3. **一貫性の維持**: モデル層のインターフェースを変更する際は、影響範囲を慎重に検討する

## まとめ

モデル層を実装することで、以下のようなメリットが得られます：

- ✅ コードの保守性が向上
- ✅ テストが書きやすくなる
- ✅ データベース実装の変更に強い
- ✅ ビジネスロジックの集約が可能
- ✅ 再利用性が向上

小規模なアプリでも、将来的な拡張性を考慮してモデル層を導入することで、長期的な保守性が向上します。
