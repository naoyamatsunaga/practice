# ポイントリセット機能（履歴移行）の実装について

本ドキュメントでは、「指定した設定時刻（24時など）になったら、ホーム画面のポイントをリセットし、以前の記録を履歴画面へ移行する」という機能をどのように実装したかの手順とアーキテクチャをわかりやすく解説します。

---

## 1. 完了の定義（満たした要件）
- [x] 設定ファイルに設定した時刻になった時、そこまでに記録したポイントは0にリセットされる。
- [x] リセット後は、記録されていたポイントは「履歴画面」から日別で確認できるようになる。
- [x] リセット時刻は、設定画面からユーザーが任意に変更できる。
- [x] 設定時刻の変更時に、何が起きるかの説明ダイアログを出す。
- [x] 設定した時刻によって、「次回の更新時刻」がいつになるのかを計算し、ホーム画面に表示する。

---

## 2. 設計のコアアイデア（最大のポイント）

この実装の最大の工夫は、**「データベースから過去のポイントを本当に削除（または操作）するわけではない」**という点です。

もし毎日24時にDBからデータを消してしまうと、履歴画面からもデータが消えてしまいます。
そこで、**「UI（画面）側の表示フィルター」** を活用した論理的なリセット手法を採用しました。

1. **現在の期間の開始時刻を計算する**
   現在時刻と「設定されたリセット時刻」を比較し、「今の期間はいつから始まったか」を算出します。
2. **ホーム画面はフィルターする**
   DBに保存されている全ポイント一覧から、「現在の期間の開始時刻」よりも**後**に記録されたデータだけをホーム画面に表示します。
3. **履歴画面はそのまま表示する**
   履歴画面ではフィルターをかけず、リセット時刻を基準にして「1日」の区切りを計算し、過去のすべてのデータを表示します。

このアプローチにより、データを安全に保持したまま、後からリセット時刻（例：00:00 → 04:00）を変更しても過去のデータが壊れない設計になっています。

---

## 3. 具体的な実装内容（ファイル別）

### ⚙️ 1. 設定・時刻計算ロジック
**対象ファイル:** `lib/view_models/settings_view_model.dart`
- 端末のローカルストレージ（`SharedPreferences`）を利用し、ユーザーが選んだ「リセット時刻（時間・分）」を保存および復元する `ResetTimeNotifier` を作成しました。
- 同ファイル内に、この機能に必須となる**純粋な時間計算関数**を配置しました。
  - `getStartOfCurrentPeriod()`: 現時刻と設定時刻から「今の期間の開始時刻」を出す。
  - `getLogicalDate()`: 「これって論理上は何日の扱いになるか？」を計算する（04:00リセット時における深夜03:00は「前日」とする計算など）。
  - `getNextResetTime()`: 次のリセット対象時刻を出す。

### 🏠 2. ホーム画面（リセットの実行）
**対象ファイル:** `lib/view_models/home_view_model.dart` / `lib/views/pages/home_page.dart`
- `homeActivityListStreamProvider` を修正し、読み込んだ全データに対して「現在の期間」に当てはまるものだけを抽出（フィルター）して返すようにしました。これにより画面上の合計値表示が自動でリセットされます。
- アプリを起動したままリセット時刻をまたいだ場合に備え、**内部でタイマー（Timer）を稼働させ、時間がきたら自動的に表示を更新（0に戻す）**仕組みを搭載しています。
- ホーム画面のUIにて、「次回リセット: 〇月〇日 〇〇:〇〇」という親切な案内テキストを追加しました。

### 📜 3. 履歴画面（データのグループ化）
**対象ファイル:** `lib/view_models/history_view_model.dart`
- 履歴画面を日ごとにまとめる際、単純なカレンダー上の「〇月〇日」ではなく、上で作った `getLogicalDate()` を使って**「設定したリセット時刻に基づく論理的な1日」**でグループ化するよう設計を変更しました。

### 🛠️ 4. 設定画面と高度な選択UI
**対象ファイル:** `lib/views/dialogs/custom_time_picker_dialog.dart` / `lib/views/pages/settings_page.dart`
- Appleのタイマーのような「ドラムロール風（スクロール式）」のピッカーと、Androidのような「文字（直接）入力式」の二つのインターフェースをシームレスに切り替えられる**カスタムダイアログ**を新規実装しました。
- 時刻設定の完了時に、「この時刻を境にホーム画面のポイントはリセットされ、以前の記録は履歴へと移動します」という確認ダイアログ（`AlertDialog`）を挟むことで、ユーザー体験を向上させています。

---

## 4. まとめ
今回の要件は、アプリの根幹となるデータの扱い方に大きく関わる部分でした。
MVVMアーキテクチャにおける「状態（Provider）」と「純粋なロジック関数」を組み合わせることで、**「データはデータベースに無傷で残りつつ、見え方だけが魔法のようにリセットされる」**という堅牢な仕組みを作り上げることができました。
